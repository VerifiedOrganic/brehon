# ADR-0006: Fjall as the only event store, Tantivy as the derived index

**Status**: Accepted
**Date**: 2026-04-14
**Deciders**: project founders

---

## Context

Brehon needs durable storage for:

- An **append-only event log** of everything that happens during a session
  (worker output, supervisor decisions, review scores, task transitions).
- **Materialized views** derived from the log (current task state,
  active leases, review-round state).
- **Run records** for completed sessions (`brehon-ports::RunStore`).
- **Proof bundles** generated by the gatekeeper
  (`brehon-ports::ProofStore`).
- **Memories, rules, and skills** that agents read and write through MCP.
- **Full-text search** over the same memories/rules/skills.

The storage layer has to:

- Persist atomically. A crash mid-write must not corrupt the log.
- Survive process restart. Recovery must reconstruct in-memory state.
- Support range scans for log replay and per-task indexes.
- Be local-first. No cloud round-trip. No daemon to install.
- Be embeddable as a Rust library. No separate server process.
- Have low startup latency. `brehon run` should not pause for seconds
  to open the store.
- Be testable. Tests must run on temp dirs without external setup.
- Be fast enough that the supervisor's event-stream reads do not
  become a bottleneck.

Search has additional requirements:

- Index memories/rules/skills for free-text query.
- Update incrementally as new entries arrive.
- Be a *derived* index — the source of truth stays in the event store,
  and the index can always be rebuilt.

The candidates considered for the event store: SQLite, fjall, redb, sled,
RocksDB, Postgres (as a server), Redis, plain JSON files.

For search: tantivy, Bleve (Go-only), Lucene (JVM-only), Meilisearch
(server), pg_trgm (server).

## Decision

**The event store, run store, and proof store are implemented exclusively
on `fjall` (an LSM-tree key-value store written in Rust). Full-text search
is provided by `tantivy`. Both are embedded libraries; neither requires a
separate process.**

Concretely:

- `crates/brehon-store-fjall/` implements `EventStore`, `RunStore`, and
  `ProofStore` (all from `brehon-ports`).
- The on-disk path is `.brehon/store/`. Schema is lexicographically-keyed
  for range-scan friendliness:
  - `log:{seq}` — global append-only log.
  - `index:agent:{agent_id}:{seq}` — per-agent index.
  - `index:task:{task_id}:{seq}` — per-task index.
  - `view:task:{task_id}` — materialized task state.
  - `queue:review:{lane}:{seq}` — per-lane review priority queue.
  - `run:{run_id}` — durable run record.
  - `proof:bundle:{bundle_id}` — proof bundle projection.
- Writes are ACID at the fjall-partition level. Atomic claim semantics
  (`acquire_task_lock`, `acquire_repo_lock`) are built on the underlying
  primitives.
- `FjallEventStore::recover()` runs on every startup. It scans the log
  and index ranges to surface orphaned leases and incomplete review
  rounds.
- There is **no in-memory mock implementation** of any of these traits
  in the workspace. Tests use real fjall on temp directories.
- `crates/brehon-search-tantivy/` implements `SearchIndex` on tantivy
  0.22. It is the derived index — losing it is recoverable by replaying
  the event store.

## Consequences

**Accepted:**

- Tests are slightly slower than they would be with an in-memory store
  because every test opens a real fjall keyspace on a temp directory.
  We accept this for the test-real-thing benefit.
- Operators cannot point Brehon at Postgres or another shared store
  without writing a new adapter. This is consistent with our local-first
  posture (see [ADR-0008](0008-local-first.md)) but constrains
  multi-machine deployments.
- Fjall is a young library compared to RocksDB or SQLite. The Rust
  ecosystem around it is smaller. We've pinned a major version and
  treat upgrades as deliberate.
- Tantivy and fjall must agree about durability semantics — the
  search index is eventually consistent with the event store. We
  document this clearly: search is a hint, not a source of truth.

**In exchange:**

- A single embedded library footprint. No daemon to install, no port
  to bind, no auth to configure.
- LSM range scans are exactly the access pattern our log replay uses.
  Recovery is one bounded scan per index range.
- ACID writes at the partition level give us atomicity for the claim
  semantics we rely on (lease acquisition, task state transitions).
- Tantivy is mature, Rust-native, and well-suited to small-to-medium
  document collections like memories/rules/skills.
- The hexagonal port lets us replace fjall with a different backend
  (sled, redb, even a hosted store) by writing a new crate. We do not
  expect to need to; the option is there if we do.

## Alternatives considered

**SQLite.** Strongly considered. It is the obvious safe choice: ubiquitous,
ACID, well-understood. Rejected for three reasons:

1. The access pattern is heavily range-scan-on-sequence (log replay,
   per-task indexes). LSM-trees are a more natural fit than B-trees
   for this pattern.
2. SQLite serializes writes globally on a single file. We expect bursty
   event-write throughput from several concurrent workers; an LSM-tree
   absorbs this more gracefully.
3. The Rust ecosystem around SQLite (rusqlite, sqlx) introduces a
   schema-migration burden we wanted to avoid. The fjall key-prefix
   schema is versionable in code without DDL.

We may revisit if fjall does not mature; nothing in the design prevents
a SQLite adapter being written.

**RocksDB.** Mature, fast, well-understood LSM. Rejected because the
RocksDB Rust bindings depend on libstdc++, complicating the single-binary
build story we wanted for `cargo install brehon`. Fjall is pure Rust.

**sled.** Considered. Rejected because sled is in a long beta and the
project signals make us cautious about staking durable state on it.

**redb.** Considered. Solid B-tree-on-mmap. Same range-scan vs. B-tree
argument applies as for SQLite, but redb is a closer fit than SQLite for
our use case. We may revisit.

**Postgres.** Considered briefly. Rejected because requiring a Postgres
server defeats the local-first commitment and is a major operational
ask of users.

**Redis.** Considered for the queue semantics. Rejected because Redis
is not durable by default and adding a Redis dependency is a major
ask of users. The fjall queue prefix gives us what we need.

**Plain JSON files.** Considered for the simplest possible thing.
Rejected because atomic claim semantics over multiple files is a
well-known source of bugs (rename atomicity differs across
filesystems, locking is hard). We tried this first and were burned
by it.

**Meilisearch / external search server** for the search role. Rejected
for the same local-first reasons as Postgres / Redis. Tantivy gives
us in-process search.
